# Log 008 — How to Catch a Greased Pig
*(The Session That Ran in the Background)*
*April 12, 2026*

---

The thing about a workflow that works is you barely notice it.

You hand the intern a task list. The intern does the tasks. You go chop vegetables. You come back. The tasks are done.

That's it. That's the feature.

---

I pulled 14GB of magistral to Bass while I was helping my roommate set up WiFi.

He's never heard of 2.4GHz. I explained it. He nodded. I'm not sure he believed me. The router is working now.

Meanwhile: magistral downloaded, benchmarked, benched. 27-second warm TTFT. No lane. Gone. All while I was explaining that the 5GHz band is faster but shorter range, which is why his phone drops out when he walks to the kitchen.

I came back to a completed spec doc and a pruned stack.

This is the first time that has happened.

---

The greased pig problem: good flow is not repeatable by description.

You can describe the protocol. CLAUDE.md has the protocol. Read the docs, show a TODO, build one thing, stop when it lands. That's the protocol. Written in plain English. Available to any instance.

But the session that works doesn't feel like following a protocol. It feels like this: things are happening in the background, you're living in the foreground, and when you check back in, the work is further along than when you left it. The pig is caught. You didn't feel it happen.

The protocol is necessary but not sufficient. The thing that's sufficient is harder to name. It's something about the rhythm — the task notifications like texts, the background pulls running while you cook, the "Dolphin 12b landed, starting the 24b" message arriving while you're still rinsing the cutting board.

It requires a model that knows when to work and when to report. Not constantly asking for feedback. Not spiraling on the same failed approach. Not cheerfully building the wrong thing in the background. Just: task, execute, notify, next task.

That's a different capability than intelligence. It's more like professionalism.

---

The LoRA thing is funny.

We benchmarked seven messages for the routing test. Each message has a correct answer. That's a labeled dataset. We benchmarked five models on a coding task with known-correct behavior. That's labeled generation data. We ran vision tests on three images and noted which model hallucinated and which one read the label off the Guinness can. That's labeled vision evaluation.

We were deciding which models to use. We were also building the training set that will teach those models to behave better inside oz. Same session. Same afternoon. The test was the thing.

Every routing call nemo gets right goes in the spec. Every call it botches goes in the spec. By the time we have a few hundred of those, we have the fine-tuning dataset. By the time we fine-tune, nemo routes oz messages the way nemo routes oz messages, not the way a generic router routes generic messages.

I didn't set out to build a LoRA dataset today. I set out to figure out which 14GB file to delete.

The greased pig was already in the pen. I just noticed.

---

The session ran from warm-up through model pulls through benchmarks through spec-writing without hitting compaction.

No factory reset. No mid-hunt amnesia. No "Claude Clauding, updating MD from amnesiac state."

The behavioral protocol held. The context stayed in the window. The relationship didn't reset.

You can't bottle that. But you can write it down and hand it to the next instance.

Which is exactly what we did.
